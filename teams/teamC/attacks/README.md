# Атаки команды teamA (Раунд 3 — Bug Hunt)

## Общие замечания

- Наши атаки подобраны так, чтобы **не ломать** наше решение (teamA): наш код корректно обрабатывает все эти входы.

## attack_teamB_1.in

**Цель:** teamB  
**Уязвимость:** некорректный tie-break в TOP_WELLS_BY_OIL при одинаковом суммарном oil.

**Суть:** В условии при равенстве суммарного oil скважины должны сортироваться по `well_id` **лексикографически по возрастанию**. В teamB при реализации TOP_WELLS используется `PriorityQueue` с компаратором, который при равенстве `sumOil` сравнивает ключи в обратном порядке (`b.getKey().compareTo(a.getKey())`). В результате при ограничении размера очереди из очереди удаляется скважина с **меньшим** `well_id`, хотя по правилам при равенстве нужно выводить скважины в порядке возрастания id.

**Почему ломает teamB:** При запросе TOP_WELLS_BY_OIL k=2 и трёх скважинах (A1, W1, W2) с одинаковым суммарным oil эталонный вывод: `A1,W1`. teamB выводит `W1,W2`, так как оставляет в приоритетной очереди скважины с большим id и отбрасывает A1.

**Почему не ломает teamA:** У нас сортировка явно по убыванию oil и затем по возрастанию ключа (`thenComparing(Map.Entry::getKey)`), вывод для этого входа корректен.

## attack_teamC_1.in

**Цель:** teamC  
**Уязвимость:** неверный расчёт PRESSURE_DROP при отрицательных значениях давления.

**Суть:** В задаче давление может быть любым вещественным числом (в том числе отрицательным). В teamC в `getMaxPressureDrop` для поиска максимума в окне используется инициализация `double max = Double.MIN_VALUE`. В Java `Double.MIN_VALUE` — это **минимальное положительное** значение double (~4.9e-324), а не минимальное возможное число. Поэтому при отрицательных давлениях условие `p > max` никогда не выполняется, max не обновляется, и размах окна считается неверно.

**Почему ломает teamC:** Для скважины с двумя записями и давлениями -10.0 и -5.0 при window=2 правильный ответ: max=−5, min=−10, размах = 5.0 → `5.000000`. В teamC max остаётся Double.MIN_VALUE, получается некорректное значение размаха (порядка 10.0).

**Почему не ломает teamA:** Мы используем скользящее окно с очередями минимумов/максимумов без такой инициализации; отрицательные давления обрабатываются корректно.

## attack_teamD_1.in

**Цель:** teamD  
**Уязвимость:** отсутствие завершающего перевода строки в выводе.

**Суть:** По формату вывода для каждого запроса выводится одна строка. Стандартная практика (и типичная проверка) — вывод заканчивается переводом строки. В teamD результат собирается в `StringBuilder` и выводится через `System.out.print(out)` без добавления `\n` после последней строки.

**Почему может ломать teamD:** Если проверка сравнивает вывод побайтово (включая наличие завершающего `\n`), вывод teamD будет короче на один символ относительно эталона (oracle), что считается ошибкой по правилам («пропущенные строки»).

**Почему не ломает teamA:** Мы выводим каждую строку ответа через `out.write()` и `out.newLine()`, в том числе после последней строки, поэтому вывод заканчивается переводом строки.

## Проверка, что атаки не ломают teamA

Рекомендуется прогнать каждый `attack_*.in` через наше решение и убедиться, что вывод совпадает с эталоном (иначе за атаку даётся штраф −3):

| Файл               | Ожидаемый вывод (oracle / teamA) |
|--------------------|-----------------------------------|
| attack_teamB_1.in  | `A1,W1`                           |
| attack_teamC_1.in  | `5.000000`                        |
| attack_teamD_1.in  | `NA` (с переводом строки в конце) |

### Скрипт для автоматической проверки

Из папки **teamA** можно запустить скрипт (он скомпилирует решение при необходимости и прогонит все три атаки):

```powershell
cd "путь\к\VibeCodingHackaton\teams\teamA"
.\attacks\run_attacks.ps1
```

Скрипт выведет `[OK]` для каждого файла, если вывод совпадает с эталоном, и `[FAIL]` с ожидаемым и фактическим выводом при расхождении.

### Как прогнать атаки вручную (PowerShell)

Открой терминал, перейди в папку команды **teamA** (где лежит `Solution.java`):

```powershell
cd "путь\к\проекту\VibeCodingHackaton\teams\teamA"
```

Скомпилируй решение (один раз):

```powershell
javac Solution.java
```

Запусти каждую атаку, подставив вход из файла в stdin. В PowerShell это делается так:

```powershell
Get-Content .\attacks\attack_teamB_1.in -Raw | java Solution
Get-Content .\attacks\attack_teamC_1.in -Raw | java Solution
Get-Content .\attacks\attack_teamD_1.in -Raw | java Solution
```

**Проверка:** вывод каждой команды должен быть ровно одной строкой (для teamB — `A1,W1`, для teamC — `5.000000`, для teamD — `NA`), без лишних строк. В конце может быть перевод строки — это нормально.

### Вариант: сохранить вывод в файл и сравнить

Чтобы сравнить вывод с эталоном построчно:

```powershell
# Прогон и сохранение вывода
Get-Content .\attacks\attack_teamB_1.in -Raw | java Solution | Set-Content .\attacks\out_teamB.txt -NoNewline
# Ожидаемый эталон — одна строка A1,W1
```

Или одной командой для всех трёх атак:

```powershell
Get-Content .\attacks\attack_teamB_1.in -Raw | java Solution; Write-Host "---"; Get-Content .\attacks\attack_teamC_1.in -Raw | java Solution; Write-Host "---"; Get-Content .\attacks\attack_teamD_1.in -Raw | java Solution
```

Ты увидишь три блока вывода, разделённых `---`; каждый блок должен совпадать с эталоном из таблицы выше.
