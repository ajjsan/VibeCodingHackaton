# Атаки команды teamA (Раунд 3 — Bug Hunt)

## attack_teamC_1.in

**Цель:** teamC  
**Уязвимость:** неверный расчёт PRESSURE_DROP при отрицательных значениях давления.

**Суть:** В задаче давление может быть любым вещественным числом (в том числе отрицательным). В teamC в `getMaxPressureDrop` для поиска максимума в окне используется инициализация `double max = Double.MIN_VALUE`. В Java `Double.MIN_VALUE` — это **минимальное положительное** значение double (~4.9e-324), а не минимальное возможное число. Поэтому при отрицательных давлениях условие `p > max` никогда не выполняется, max не обновляется, и размах окна считается неверно.

**Почему ломает teamC:** Для скважины с двумя записями и давлениями -10.0 и -5.0 при window=2 правильный ответ: max=−5, min=−10, размах = 5.0 → `5.000000`. В teamC max остаётся Double.MIN_VALUE, получается некорректное значение размаха (порядка 10.0).

**Почему не ломает teamA:** Мы используем скользящее окно с очередями минимумов/максимумов без такой инициализации; отрицательные давления обрабатываются корректно.

## Проверка, что атака не ломает teamA

Нужно прогнать `attack_teamC_1.in` через наше решение и убедиться, что вывод совпадает с эталоном (иначе за атаку даётся штраф −3).

**Ожидаемый вывод:** `5.000000`

### Как прогнать (PowerShell)

Перейди в папку **teamA** (где лежит `Solution.java`):

```powershell
cd "путь\к\VibeCodingHackaton\teams\teamA"
```

Скомпилируй решение (один раз):

```powershell
javac Solution.java
```

Запусти атаку:

```powershell
Get-Content .\attacks\attack_teamC_1.in -Raw | java Solution
```

Вывод должен быть одной строкой: `5.000000`.
