# 🛢 Production Log Aggregator — Раунд 4 (Мутация требований)

## Контекст

В этом раунде изменяется семантика одного из запросов.

Все остальные требования (Раунд 1 + Раунд 2) остаются в силе:
- формат ввода/вывода не меняется
- ограничения на 10^7 строк сохраняются
- потоковая обработка обязательна

Код можно изменять.

---

## 🔁 Изменение требования

Изменяется логика запроса:

PRESSURE_DROP well_id=XXX window=W

---

### ❌ Было (Раунд 2)

Для каждого окна из W подряд идущих записей:

max(pressure) − min(pressure)

Ответ — максимум этого значения по всем окнам.

---

### ✅ Стало (Раунд 4)

Для каждого окна из W подряд идущих записей:

Нужно найти **максимальное падение давления во времени** внутри окна.

Формально:

max(pressure_i − pressure_j),  
где записи i и j находятся внутри одного окна  
и i < j (то есть j идёт позже во входе)

Важно:

- учитываются только падения вперёд во времени
- если давление только растёт — результат 0
- если в окне меньше W записей — окно не рассматривается

---

## 📌 Граничные случаи

- Если `W <= 0` → вывести `NA`
- Если скважина отсутствует → вывести `NA`
- Если число валидных записей < W → вывести `NA`

---

## ⚙️ Ограничения производительности

- DATA может содержать до 10^7 строк
- Решение должно быть потоковым
- Нельзя хранить все записи целиком
- Для обработки окна допускается хранить только данные, необходимые для текущего окна

---

## 📤 Формат вывода

- Вещественные значения — 6 знаков после запятой
- `NA` — в тех же случаях, что и ранее
- Порядок вывода строго соответствует порядку запросов

---

## 🎯 Цель раунда

Адаптировать существующую архитектуру к новой логике.

Наивная реализация через:

max(pressure) − min(pressure)

больше не корректна.
