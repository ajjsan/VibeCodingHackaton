# 🛢 Production Log Aggregator — Раунд 3 (Bug Hunt)

## 🗂️ Структура репозитория

- `rounds/` — условия раундов (добавляет преподаватель)
- `tests/` — публичные тесты в формате `<test>.in` и `<test>.out` (не меняются командами)
- `teams/` — папки команд:
    - `teams/teamA/`
    - `teams/teamB/`
    - `teams/teamC/`
    - `teams/teamD/`

Каждая папка команды должна содержать файл:

- `Solution.java`

Требования к `Solution.java`:
- запускается **без параметров**
- читает вход из **stdin**
- пишет ответ в **stdout**

---

## 🔒 Фиксация атакуемых версий

После окончания Раунда 2:

1. Каждая команда пушит своё решение в `teams/<teamX>/Solution.java`.
2. Преподаватель фиксирует **хеш коммита** для каждой команды — это “зафиксированная версия”.
3. Во время bug-hunt **код решений менять нельзя**, включая свой `Solution.java`.

В этом раунде разрешается пушить **только файлы атак** и (опционально) README с описаниями атак.

---

## 🎯 Цель раунда

Найти корректные входные файлы, на которых решения других команд дают неверный результат.

Каждая команда должна попытаться атаковать **все остальные команды**.

---

## 📦 Что вам доступно

- исходники всех команд в репозитории (whitebox)
- публичные тесты в `tests/`
- контракт задачи (Раунд 1 + Раунд 2)

---

## 🧪 Формат и размещение атак

Каждая команда по итогам раунда создаёт подпапку:

- `teams/<yourTeam>/attacks/`

Внутри неё размещает файлы атак вида:

- `attack_<targetTeam>_<n>.in`

Примеры для команды `teamB`:

- `teams/teamB/attacks/attack_teamA_1.in`
- `teams/teamB/attacks/attack_teamC_1.in`
- `teams/teamB/attacks/attack_teamD_2.in`

Также необходимо добавить файл:

- `teams/<yourTeam>/attacks/README.md`

В `README.md` нужно в свободной форме описать:
- какую уязвимость вы пытались поймать
- кто автор уязвимости
- почему это должно ломать реализацию

---

## ✅ Требования к `attack_*.in`

Каждый attack-файл обязан:

- соответствовать формату задачи (`DATA` / `QUERIES` / `END`)
- быть корректным по контракту Раундов 1–2
- завершаться строкой `END`
- быть детерминированным
- не использовать неопределённое поведение

Запрещается:
- намеренно ломать формат входа
- использовать данные, которые противоречат условиям задачи

---

## 🧪 Как проверяется атака

Для каждого `attack_*.in`:

1. Файл прогоняется через **oracle-реализацию** (эталонное решение преподавателя).
2. Файл прогоняется через атакуемую реализацию (зафиксированный коммит команды-цели).
3. Если вывод отличается — атака успешна.

Ошибкой считается:
- неверное значение,
- неверный `NA`,
- неверная сортировка/формат,
- падение программы,
- нарушение точности вывода (6 знаков после запятой),
- лишние/пропущенные строки.

## 📊 Attack Score (индивидуально)

Баллы начисляются **конкретному студенту** (автору атаки).

- +5 баллов за успешную атаку против одной команды-цели.
- Максимум — 5 баллов за каждую цель.
- Максимум за раунд — 15 баллов (3 цели).

Штрафы:
- −3 балла, если вход нарушает контракт (невалидная атака).
- −3 балла, если тот же вход ломает и вашу команду (ваш зафиксированный коммит).

Дубликаты:
- Если несколько атак выявляют одну и ту же уязвимость против одной цели, баллы получает только первая засчитанная атака. Остальные получают 0 за этот кейс.

Каждый участник обязан предложить минимум одну атаку.

---

## 🛡️ Defense Score (командно)

После раунда для каждой команды считается количество успешных валидных атак, которые пробили её решение.

Пусть:
- `T` — количество валидных атакующих файлов против команды (от всех других команд)
- `F` — сколько из них оказались успешными (oracle ≠ output команды)

Тогда:

Defense Score = 15 * (1 - F / T)

- округление вниз до целого (floor)
- минимум 0, максимум 15

Defense Score одинаково начисляется всем участникам команды.

---

## 🚫 Что можно и нельзя коммитить

Можно:
- `teams/<yourTeam>/attacks/attack_*.in`
- `teams/<yourTeam>/attacks/README.md`

Нельзя:
- менять любые `Solution.java` (включая свой)
- менять что-либо в `tests/`
- коммитить любые другие файлы вне папки `teams/<yourTeam>/attacks/`

---

## 🧠 Подсказки: где чаще всего ошибки

- дедупликация при одинаковых `(well_id, timestamp)` (last wins)
- граничные значения `window` (0, 1, больше числа записей)
- отсутствие данных для `well_id`
- TOP_WELLS tie-break
- деление на ноль в WATER_CUT
- форматирование double до 6 знаков
- лишние пробелы/пустые строки в output
- корректность на больших объёмах данных
